# Core Concepts

Crucible organizes scientific data around four types of objects. Understanding how they relate makes it easier to structure your data.

---

## Projects

A **project** is the top-level organizational unit. Think of it as roughly equivalent to a user proposal or research initiative at a user facility.

Projects serve two purposes:

1. **Grouping** — datasets and samples belong to a project, making them easier to find and filter.
2. **Access control** — all members of a project can read the datasets and samples within it. Adding a user to a project grants them access to all associated data.

Every dataset and sample must belong to a project. The `project_id` is a short, human-readable identifier chosen at creation time (e.g., `MFP12345`).

[Project model →](models/project.md) | [Project management guide →](user-guide/projects.md)

## Datasets

A **dataset** is the core data object in Crucible. It can combine:

- **Files** — the actual measurement data (optional; a dataset record can exist without any attached files)
- **Structured metadata** — measurement type, instrument, session, timestamp, format
- **Scientific metadata** — free-form key-value pairs for experiment-specific parameters (e.g., `temperature_K`, `beam_energy_eV`)
- **Keywords** — searchable tags

Datasets can be linked to each other in parent-child relationships to represent processing pipelines (e.g., raw data → calibrated → analyzed).

[Dataset model →](models/dataset.md) | [Working with datasets →](user-guide/datasets.md)

---

## Samples

A **sample** represents a physical or computational material that data was collected from. Samples are useful for tracking provenance: which datasets were measured from which material, and how that material was prepared.

Like datasets, samples support parent-child hierarchies. A bulk crystal, a cleaved wafer cut from it, and a thin film deposited on the wafer can all be modeled as a parent-child sample chain.

A dataset can be linked to one or more samples, and a sample can be linked to one or more datasets.

[Sample model →](models/sample.md) | [Working with samples →](user-guide/samples.md)

## Instruments

An **instrument** represents the physical equipment from which a dataset originated. Linking a dataset to an instrument (via `instrument_name`) provides a consistent way to filter and search data by the equipment that produced it.

Instruments are shared across projects and exist independently of any single project. However, each distinct physical instrument should have its own entry — if two organizations each own a Verios SEM, those are two separate instruments in Crucible, not one shared record. The distinguishing information (owner, location, description) is what makes each entry unique.

If an instrument already exists in Crucible, reference it by name in your datasets. If not, create it first with `client.instruments.create()` — the method returns the existing record if one with that name already exists.

[Instrument model →](models/instrument.md)

## How they relate

```
Project
├── Dataset (measurement=XRD, instrument=Beamline 12.3.2)
│   ├── parent Dataset (raw data)
│   └── linked Sample → Silicon wafer A
│       └── parent Sample → Silicon boule #3
└── Dataset (measurement=SEM imaging)
    └── linked Sample → Silicon wafer A
```

A project contains datasets and samples. Datasets are linked to the samples they were measured from. Both datasets and samples can form hierarchies. Instruments are referenced by datasets but exist independently of any single project.

---

## IDs

Every object in Crucible has a system-assigned `unique_id` generated by the [`mfid`](https://pypi.org/project/mfid/) package. The `unique_id` is the stable identifier you use to retrieve, update, link, and download objects.
